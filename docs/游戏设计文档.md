## 一、游戏概览

### 1.1 一句话描述
一款**单人对战 3 个 AI 的竞拍策略游戏**。玩家在 5 轮竞拍中与 AI 争夺装满藏品的仓库，每轮有不同的成交规则。玩家需要在信息不完整的矩阵中判断仓库价值，合理出价，以最低成本获取最高价值的藏品。

### 1.2 核心体验
- **赌博心理**：你不知道仓库里有什么，但你需要出价
- **博弈对抗**：3 个 AI 会根据你的出价动态调整策略
- **信息博弈**：通过道具和技能获取有限的仓库信息
- **经济循环**：赚取哈夫币 → 购买道具 → 获取更多信息 → 更精准出价
- **藏品展出**：将赢得的藏品放入展厅，每轮自动赚取哈夫币，实现资产增值

### 1.3 技术栈
| 类别 | 技术 | 说明 |
|------|------|------|
| 前端 | 纯 HTML + CSS + JavaScript | 单文件 SPA，无框架依赖 |
| 音频 | Web Audio API | 程序化生成音效，无音频文件 |
| 后端 | Python http.server | 仅用于本地开发静态文件服务 |
| 构建 | 无 | 直接打开 HTML 文件即可运行 |

### 2.2 游戏状态对象（gameState）

这是整个游戏的核心数据结构，修改任何系统前必须先理解它：

```javascript
gameState = {
  currentRound: 1,        // 当前轮次（1~5）
  maxRounds: 5,           // 最大轮次
  money: 3000000,          // 玩家当前资金（初始 300 万）
  aiMoney: [2500000, 1800000, 3200000],  // 3 个 AI 的资金
  hafCoins: 0,             // 玩家的哈夫币（二级货币）
  collectedItems: [],      // 玩家已收集的所有藏品
  collectedValues: [],     // 对应的藏品价值
  isBidding: false,        // 是否正在出价结算中（锁定状态）
  bidAmount: 0,            // 玩家当前输入的出价金额
  roundHistory: [],        // 历史轮次记录（当前未使用）
  playerCharId: 'ethan',   // 玩家选择的角色 ID
  gridItems: [],           // 仓库矩阵 60 格的物品数据
  hasSecurityGuard: false, // 是否购买了保安服务（单局生效）
  hasInsurance: false      // 是否购买了保险服务（单局生效）
}
```

**⚠️ 关键约束**：
- `isBidding = true` 时，所有出价操作被锁定
- 玩家资金不足时不能出价
- 每轮只能使用 1 个道具（由 `roundItemUsed` 控制）

## 三、竞拍核心规则

### 3.1 五轮成交规则

这是游戏最重要的机制。每一轮有不同的"成交条件"——第一名出价必须满足与第二名的倍数关系才能成交：

| 轮次 | 规则名称 | 成交条件 | 成交价计算 | 策略含义 |
|------|----------|----------|------------|----------|
| 第 1 轮 | 翻倍成交 | 1st ≥ 2nd × 2.0 | 2nd × 2.0 | 必须大幅领先才能成交 |
| 第 2 轮 | 1.8倍成交 | 1st ≥ 2nd × 1.8 | 2nd × 1.8 | 仍然需要显著领先 |
| 第 3 轮 | 1.5倍成交 | 1st ≥ 2nd × 0.5 | 2nd × 0.5 | 只需达到对方一半即可 |
| 第 4 轮 | 1.2倍成交 | 1st ≥ 2nd × 0.2 | 2nd × 0.2 | 只需对方的五分之一 |
| 第 5 轮 | 加1元成交 | 1st > 2nd（特殊） | 2nd + 1 | 只要比第二名多 1 元 |

### 3.2 成交判定逻辑

```
输入：allBids（按出价从高到低排序的所有出价）
      winner = allBids[0]（第一名）
      second = allBids[1]（第二名）
      rule = ROUND_RULES[currentRound - 1]

判定：
  if rule.special（第 5 轮）:
      conditionMet = true  // 第 5 轮总是成交
      dealPrice = second.val + 1
  else:
      conditionMet = (winner.val >= second.val * rule.threshold)
      dealPrice = second.val * rule.threshold（向下取整）

结果：
  if conditionMet:
      → 扣除买家 dealPrice 资金
      → 买家获得 20 件随机藏品
      → 显示结算弹窗
  else:
      → 本轮流拍
      → 进入下一轮
      → 重置出价状态
```

### 3.3 成交价计算示例

假设 4 人出价：你 ¥600,000、AI1 ¥400,000、AI2 ¥300,000、AI3 ¥200,000

| 轮次 | 规则 | 1st=¥600K, 2nd=¥400K | 条件检查 | 成交价 |
|------|------|----------------------|----------|--------|
| 1 | ×2.0 | 600K ≥ 400K×2.0=800K? | ❌ 不成交 | - |
| 2 | ×1.8 | 600K ≥ 400K×1.8=720K? | ❌ 不成交 | - |
| 3 | ×0.5 | 600K ≥ 400K×0.5=200K? | ✅ 成交 | ¥200,000 |
| 4 | ×0.2 | 600K ≥ 400K×0.2=80K? | ✅ 成交 | ¥80,000 |
| 5 | +1 | 600K > 400K? | ✅ 成交 | ¥400,001 |

**⚠️ 注意**：成交价总是基于第二名的出价计算，不是基于第一名的出价！这意味着出价过高不会增加成交价，但可能确保成交。

## 四、仓库矩阵系统

### 4.1 矩阵结构
- **布局**：6 列 × 10 行 = **60 个格子**
- **CSS Grid**：`grid-template-columns: repeat(6, 1fr)`
- 每个格子是一个 `wh-cell` 元素

### 4.2 物品生成规则

每个格子在游戏初始化时通过 `pickItem()` 生成：

```javascript
function pickItem() {
  // 随机选择 8 种藏品图片之一
  const base = collectionItems[randomIndex];
  
  // 随机决定稀有度（6 种可能）
  const rarity = ['normal','normal','normal','normal','gold','red'][randomIndex];
  
  // 根据稀有度生成价值
  if (rarity === 'gold')   value = 150000 + random(0~200000);  // 15万~35万
  if (rarity === 'red')    value = 80000 + random(0~100000);   // 8万~18万
  if (rarity === 'normal') value = 10000 + random(0~40000);    // 1万~5万
  
  return { src, rarity, value };
}
```

### 4.3 稀有度分布
| 稀有度 | 概率 | 价值范围 | 视觉标识 |
|--------|------|----------|----------|
| normal（普通） | 4/6 ≈ 66.7% | ¥10,000 ~ ¥50,000 | 无特殊标记 |
| gold（金色） | 1/6 ≈ 16.7% | ¥150,000 ~ ¥350,000 | 金色边框 + 发光 |
| red（红色） | 1/6 ≈ 16.7% | ¥80,000 ~ ¥180,000 | 红色边框 + 发光 |

### 4.4 藏品图片资源
共 8 张图片：`image/藏品/1.png` ~ `image/藏品/8.png`，在结算弹窗中展示。图片本身不携带价值信息，价值由 `pickItem()` 随机生成。

## 五、道具系统

### 5.1 道具类型

| ID | 名称 | 图标 | 效果 | 获取方式 |
|----|------|------|------|----------|
| 1 | 扫描器 | 🔍 | 揭示仓库矩阵中随机 3 格的稀有度 | 初始分配 / 商城购买 |
| 2 | 透视镜 | 👁 | 随机抽检 2 件物品（仅玩家可见价值） | 初始分配 / 商城购买 |
| 3 | 鉴定眼 | 🔎 | 揭示仓库矩阵中随机 5 格的稀有度 | 初始分配 / 商城购买 |

### 5.2 使用规则
- 每轮每人**只能使用 1 个道具**（`roundItemUsed` 标记）
- 玩家通过**数字键 1-5** 或**点击物品栏**使用
- 使用前会弹出确认框
- AI 在每轮开始时自动使用 1 个随机道具（效果不可见）

### 5.3 物品栏结构
- 每个玩家有 **5 个格子**
- 初始时随机填充 5 个道具
- 已使用的道具显示为半透明
- 商城购买的道具会**替换已使用的格子**

### 5.4 揭示效果实现
当使用扫描器/鉴定眼时：
1. 从 60 格中随机选取 N 个不重复的格子
2. 读取 `gameState.gridItems[idx].rarity`
3. 在格子上显示稀有度标签（金/稀/普）
4. 在日志中添加技能消息

当使用透视镜时：
1. 调用 `pickItem()` 生成 N 个新物品（**注意：这不是读取矩阵中的真实物品**）
2. 在日志中展示物品的稀有度和价值
3. 仅对玩家本人可见

**⚠️ 已知设计问题**：透视镜的"抽检"实际上是生成新随机物品，而非读取仓库矩阵中的真实物品。这是一个可改进的设计点。

## 六、哈夫币与商城系统

### 6.1 哈夫币获取
- 玩家赢得竞拍后，利润（藏品总价值 - 成交价）按 **1/1000** 比例转化为哈夫币
- 公式：`hafCoins += Math.max(0, Math.floor((totalVal - dealPrice) / 1000))`
- 如果亏损（利润为负），不获得哈夫币

### 6.2 商城商品

| 商品 | 价格（哈夫币） | 说明 |
|------|----------------|------|
| 扫描器 | 50 | 替换一个已使用的道具格 |
| 透视镜 | 80 | 替换一个已使用的道具格 |
| 鉴定眼 | 120 | 替换一个已使用的道具格 |

### 6.3 购买条件
- 哈夫币余额 ≥ 商品价格
- 物品栏中必须有**已使用**的空位（5 个格子全部未使用时无法购买）
- 购买后替换第一个已使用的格子

## 十·三、安保服务系统（保安+保险）

### 10C.1 功能概述
大厅界面提供两种安保服务，每局开始前可购买/退订，**单局生效**，下一局需重新购买。

### 10C.2 保安服务
- **价格**：¥200,000 / 局
- **效果**：失窃概率 -8%，失火概率 -6%
- **基础概率**：失窃 20%，失火 15%
- **购买后概率**：失窃 12%，失火 9%
- **退订**：点击已购服务可退订，全额返还

### 10C.3 保险服务
- **价格**：¥300,000 / 局
- **效果**：失窃事件赔付损失的 60%，失火事件赔付损失的 80%
- **退订**：点击已购服务可退订，全额返还

### 10C.4 购买入口
大厅右侧面板底部的「安保服务」区域，显示两个可点击的服务项。资金不足时提示错误。

## 十·五、随机事件系统

### 10E.1 功能概述
游戏结束时（点击「开始下一局」或「重新开始」），若玩家有藏品，触发随机风险事件检查。

### 10E.2 事件类型

| 事件 | 基础概率 | 效果 | 保安减少 |
|------|----------|------|----------|
| 失窃 | 20% | 损失 20% 已收集藏品 | -8%（降至12%） |
| 失火 | 15% | 损失 15% 已收集藏品 | -6%（降至9%） |

### 10E.3 保险赔付
- 失窃赔付：损失价值 × 60%
- 失火赔付：损失价值 × 80%
- 赔付金额直接加入玩家资金

### 10E.4 事件流程
1. 结算弹窗关闭
2. 弹出「风险评估」弹窗
3. 显示保安状态和实际概率
4. 显示事件结果（失窃/失火/安全）
5. 如有保险赔付，显示赔付金额
6. 玩家点击确认后，损失生效，进入下一局/大厅

### 10E.5 触发条件
- 仅在玩家有藏品时触发（无藏品则跳过）
- 失窃和失火独立判定，可能同时发生
- 两个事件的损失物品不重叠

### 7.2 AI 出价算法

```
函数 generateAIBids(isPreliminary):
  aggression = 0.3 + currentRound × 0.1   // 侵略性随轮次递增

  对每个 AI：
    if isPreliminary（预出价）或 玩家未出价：
      aiBid = maxBid × (0.15 + random × 0.35)   // 随机出 15%~50%
    else：
      ratio = 0.7 + random × aggression           // 出玩家出价的 70%~(70%+aggression)%
      aiBid = playerBid × ratio
      if aiBid > maxBid：
        aiBid = maxBid × 0.9                      // 不超过自身 90%
    
    aiBid = max(aiBid, 50000)                     // 最低出价 5 万
```

### 7.3 AI 行为特征
- **侵略性递增**：第 1 轮 aggression=0.4，第 5 轮 aggression=0.8
- **资金上限保护**：永远不会出超过自身资金 90% 的价格
- **最低保障**：最低出价 ¥50,000
- **物品使用**：每轮随机使用 1 个道具（效果对玩家不可见）
- **无策略性**：当前 AI 没有"瞄准特定稀有度"或"读取矩阵"的逻辑

## 九、结算系统

### 9.1 结算触发条件
- 玩家或 AI 赢得竞拍且满足成交条件
- 第 5 轮自动成交（特殊规则）

### 9.2 结算流程
1. 计算成交价
2. 扣除买家资金
3. 生成 **20 件**随机藏品
4. 计算藏品总价值
5. 如果是玩家获胜：
   - 将 20 件藏品加入 `gameState.collectedItems`
   - 计算并添加哈夫币
6. 显示结算弹窗：
   - 左侧：成交价、藏品总估值、利润/亏损
   - 中间：20 件藏品的网格展示（逐个揭示动画）
   - 右侧：拍卖师形象

### 9.3 藏品揭示动画
- 每个格子先显示加载圆环（SVG 动画）
- 圆环走完后显示藏品图片
- 间隔 350ms 逐个揭示
- 金色/红色物品有特殊边框发光效果
- 伴随对应品质的音效

### 9.4 利润计算
```
利润 = 藏品总价值 - 成交价
如果 利润 ≥ 0：显示绿色 "+¥XX"
如果 利润 < 0：显示红色 "-¥XX"
```

## 十一、音频系统

### 11.1 音效列表

| 音效 | 触发时机 | 实现方式 |
|------|----------|----------|

- 噪声通过 `AudioBuffer` 生成随机采样
- 所有音效均为程序化合成，无外部音频文件
- 音频上下文在首次交互时创建（`AudioContext`）
